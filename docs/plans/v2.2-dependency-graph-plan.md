# Plan: kern review v2.2 — Dependency-Aware Review + LLM Context Pipeline

## The Pitch

> "We don't need 5000 rules. We need 26 real ones + an LLM that actually understands your code."

Three layers, each doing what it's best at:

1. **kern review** — 26 AST rules catch structural bugs mechanically (zero FPs, instant)
2. **kern IR** — compress the code 5x so the LLM gets more context per dollar
3. **LLM** — catches the nuanced stuff (bad naming, wrong abstraction, business logic bugs)

SonarQube tries to do everything with regex → 200 rules, 180 noise. We let each layer do what it's best at.

v2.2 makes this real by giving the LLM the **full import context**, not just one file in isolation.

---

## Problem

`kern review` currently analyzes files in isolation. When `toast.ts` imports `useStore` from `store.ts` and `store.ts` has a memory leak, we miss it. The `--llm` mode sends one file at a time — the LLM can't see cross-file patterns.

---

## Design

### Core Concept: "Changed files + their import graph"

```
Changed files (from --diff or explicit paths)
       ↓
Resolve import graph (transitive closure)
       ↓
Run ALL 26 rules on ALL files in graph
       ↓
Filter output:
  - Changed file findings → error/warning (normal)
  - Upstream dependency findings → info only ("context finding")
       ↓
--llm mode: send KERN IR for ALL files in graph (LLM gets full context)
```

### Finding Classification

| Origin | Severity | Label | Show by default |
|--------|----------|-------|-----------------|
| Changed file | error/warning/info | Normal finding | Yes |
| Direct import (distance=1) | info | `[upstream]` | Yes |
| Transitive import (distance>1) | info | `[deep-upstream]` | Only with `--verbose` |

### New CLI Flags

```bash
kern review src/stores/toast.ts --graph          # Resolve imports, review full graph
kern review --diff origin/main --graph           # Changed files + their imports
kern review src/ -r --batch                      # Batch mode for large repos
kern review src/ --graph --llm                   # Full context LLM review
kern review src/ --graph --max-depth=3           # Cap traversal depth
```

## Part A: Import Graph Resolver

### A1. `resolveImportGraph(entryFiles, options)` in `packages/review/src/graph.ts`

**Input:** `string[]` (file paths) + options (`maxDepth`, `excludeNodeModules`, `tsconfigPath`)

**Algorithm:**
1. Create a single ts-morph `Project` (reuse `createProject()` from inferrer)
2. Add entry files to the project
3. For each file, walk `ImportDeclaration` nodes via AST
4. Resolve each import specifier to absolute path using ts-morph's module resolution
5. Skip: `node_modules`, `.d.ts`, non-TS files
6. Track distance from entry files (BFS, not DFS — avoids circular dep recursion)
7. Cap at `maxDepth` (default: 5) to bound large repos
8. Return `GraphResult`:

```typescript
interface GraphResult {
  /** All files in the graph, including entry files */
  files: GraphFile[];
  /** Entry files (the changed files) */
  entryFiles: Set<string>;
  /** Total files resolved */
  totalFiles: number;
  /** Files skipped (node_modules, .d.ts, etc.) */
  skipped: number;
}

interface GraphFile {
  filePath: string;
  distance: number;          // 0 = entry file, 1 = direct import, etc.
  importedBy: string[];      // which files import this one
  imports: string[];          // what this file imports
}
```

### A2. Gotchas to Handle

- **Path aliases** (`@/`, `~/`, `#/`) — read tsconfig `paths` via ts-morph's compilerOptions
- **Barrel files** (`index.ts` re-exports) — resolve through, don't stop at barrel
- **Circular imports** — BFS with visited set handles naturally
- **Monorepo workspace imports** — resolve `@kernlang/core` to local package via pnpm workspace
- **Dynamic imports** — skip for v2.2, flag as info ("dynamic import not followed")

## Part B: Graph-Aware Review Pipeline

### B1. Modify `reviewSource` / `reviewFile` to accept graph context

Add optional `graphContext` parameter:

```typescript
interface GraphContext {
  distance: number;           // 0 = changed file, 1+ = upstream
  entryFile: boolean;         // is this a changed file?
  importedBy: string[];       // who imports this file
}
```

### B2. Finding Provenance

Every `ReviewFinding` gets new optional fields:

```typescript
interface ReviewFinding {
  // ... existing fields ...
  origin?: 'changed' | 'upstream';
  distance?: number;
  importedBy?: string[];
}
```

### B3. Reporter Filtering

In `formatReport()`:
- `origin === 'changed'` → show normally (error/warning/info)
- `origin === 'upstream'` && `distance === 1` → show as info with `[upstream]` tag
- `origin === 'upstream'` && `distance > 1` → hide unless `--verbose`

### B4. LLM Mode Enhancement

`buildLLMPrompt()` with graph context:
- Include KERN IR for ALL files in graph (not just changed files)
- Mark each node with `[CHANGED]` or `[CONTEXT d=N]` so LLM knows what to focus on
- LLM instruction: "Focus findings on [CHANGED] nodes. Reference [CONTEXT] nodes only to support your analysis."

## Part C: Batch Mode for Large Repos

### C1. `--batch` flag

For repos with 100+ files:
1. Collect all TS files (existing `collectTsFiles`)
2. Group into batches of 20 files (configurable with `--batch-size=N`)
3. Run review pipeline per batch with shared Project (memory optimization)
4. Merge findings across batches, dedup
5. Progress output: `  Batch 1/5: 20 files reviewed (3 findings)`

### C2. Memory Management

- Reuse single ts-morph `Project` across batches (don't create new one per file)
- Release SourceFile references after each batch
- Cap max files at 500 (configurable) — warn if exceeded
