**Date:** 2026-03-22
**Author:** Claude + Raphael + Codex (validation)
**Status:** Draft
**Target:** @kernlang/udr v1.0.0 (kern v4.0.0)

---

## 1. Why UDR — Kern is a Specification Compiler

Kern is NOT a program compiler (like LLVM, Svelte, TypeScript) and NOT a transpiler (like Babel, Sass). It is a **specification compiler** — it expands compact structural specifications into verbose framework implementations.

| System | Input | Output | Type |
|--------|-------|--------|------|
| LLVM | C program | Machine code | Program compiler |
| Svelte | Component (template+script+style) | JavaScript | Hybrid (spec+program) compiler |
| TypeScript | TS program | JS program | Transpiler |
| **Kern** | **Structural specification** | **Framework code** | **Specification compiler** |
| OpenAPI codegen | API spec | REST client | Specification compiler |
| Prisma | Schema | ORM client | Specification compiler |

Kern's closest analogs are NOT LLVM or Svelte — they're **OpenAPI codegen, Prisma generators, and GraphQL codegen**. All specification compilers use the same pattern: understand the spec → decide the mapping → render the code.

This is WHY UDR works and WHY it's simple:
- **No type inference** — specs declare types explicitly
- **No control flow analysis** — specs don't have control flow (handler blocks are opaque)
- **No optimization passes** — the output IS the implementation
- **No execution semantics** — specs describe STRUCTURE, not BEHAVIOR

The D layer can be data (~95%) because spec-to-code mapping IS a data problem. "How does 'reactive state' look in Svelte?" is a pattern string, not a semantic transformation.

**Handler blocks (`<<<>>>`) are the exception.** They contain PROGRAMS, not specs. But they're OPAQUE — Kern passes them through. The target language's compiler handles them.

---

## 2. The Pattern

Every multi-target specification compiler converges to three layers:

```
U — UNDERSTAND    Shared analysis. Extract semantic meaning from AST.
D — DECIDE        Per-target manifest. Declare HOW to render each concern.
R — RENDER        Shared engine. Execute decisions against the analyzed tree.
```

UDR is more specific than generic "frontend/IR/backend":
- **U** produces a defined contract (`AnalyzedComponent`) — not just "an IR"
- **D** is declarative data (~95%) — not "backend code"
- **R** is ONE shared engine with strategy switches — not "one backend per target"

**Validated against:** LLVM, Svelte, Haxe, Fable, Nim, JVM. All follow UDR.

**Budget:**
- U: ~1,100 LOC (shared, run once per compilation)
- D: ~80-100 LOC per target (data manifests)
- R: ~650 LOC (shared engine, all targets)
- Plugins: ~350 LOC (Tailwind classifier, Python language)
- **Total: ~3,700 LOC** for 12 targets (vs current ~10,000 = **~63% reduction**)

### 2.4 decomposeTree — The Core Function

Converts IRNode tree to AnalyzedChild[]. This is where Kern-specific node types disappear and generic child types emerge.

```typescript
type AnalyzedChild =
  | { kind: 'node'; node: AnalyzedNode }
  | { kind: 'conditional'; test: string; then: AnalyzedChild[]; else?: AnalyzedChild[] }
  | { kind: 'loop'; collection: string; item: string; key?: string; body: AnalyzedChild[] }
  | { kind: 'raw'; code: string };
```

**Filtering rules:**
- `state`, `theme`, `handler`, `on`, `emit`, `prop`, `logic`, `metadata`, `fetch` → SKIP (already collected)
- `conditional` → AnalyzedChild.conditional
- `tabs` → decomposed to state + buttons + conditionals
- `list` with `items` prop → AnalyzedChild.loop
- Everything else → AnalyzedChild.node

After decomposeTree, the R layer NEVER sees Kern node types. Only: `node`, `conditional`, `loop`, `raw`.

## 3. Layer D — Decide

### 3.1 TargetDecisions Type

**Delimiter: `%name%`** (not `{{name}}` — avoids Vue/Svelte template syntax collisions).

```typescript
interface TargetDecisions {
  id: string;
  family: 'ui' | 'server';

  // FILE ASSEMBLY
  file: {
    extension: string;
    assembly: SfcAssembly | SingleFunctionAssembly | ModuleAssembly;
  };

  // ELEMENT RENDERING
  classAttr: 'className' | 'class';

  // STATE
  state: {
    pattern: string;             // 'let %name% = $state(%init%);'
    genericPattern?: string;     // 'let %name% = $state<%type%>(%init%);'
    importSource?: string;       // 'react' | 'vue' | undefined
    importNames?: string[];      // ['useState'] | ['ref']
  };

  // EVENTS
  events: {
    pattern: string;             // 'on%event%={%handler%}'
    casing: 'lowercase' | 'camelCase';
    expressionWrap: string;      // '() => { %code% }' or '%code%'
  };

  // BINDING
  binding:
    | { mode: 'native'; pattern: string }
    | { mode: 'controlled'; valuePattern: string; changePattern: string };

  // CONDITIONAL (3 strategies)
  conditional:
    | { mode: 'wrap_block'; open: string; elseOpen: string; close: string }
    | { mode: 'decorate_child'; attr: string; elseAttr: string; wrapperTag: string }
    | { mode: 'expr_wrapper'; andPattern: string; ternaryPattern: string };

  // LOOP (3 strategies)
  loop:
    | { mode: 'wrap_block'; open: string; close: string }
    | { mode: 'decorate_child'; attr: string }
    | { mode: 'expr_map'; pattern: string };

  // STYLES
  styles:
    | { mode: 'scoped_css' }
    | { mode: 'utility_classes' }
    | { mode: 'inline_object' }
    | { mode: 'rn_stylesheet' };

  // TEXT
  text: { interpolation: string };  // '{%expr%}' or '{{ %expr% }}'

  // CONDITIONAL CLASS
  conditionalClass: { pattern: string };

  // DYNAMIC PROPS
  dynamicProp: { pattern: string };

  // HANDLER REWRITING
  handlerRewrite?: {
    stateSetterReplace?: (name: string, expr: string) => string;
    emitRewrite?: (name: string) => string;
  };

  // WINDOW EVENTS
  windowEvent?: { pattern: string };

  // IMPORTS
  imports: { autoImported: string[] };

  // COMPONENT REFS
  componentRef: { importPattern: string; propsShorthand: boolean };

  // HEAD/METADATA
  head?: { open: string; close: string; titlePattern: string; metaPattern: string }
       | { mode: 'metadata_export'; pattern: string }
       | { mode: 'composable'; pattern: string };

  // ROUTING (meta-frameworks only)
  routing?: {
    inferPath: (name: string) => string;
    pagePattern: (route: string) => string;
    layoutPattern: (route: string) => string;
    serverPattern: (route: string, method: string) => string;
    loadPattern?: (route: string) => string;
  };
}
```

### 4.3 interpolate()

```typescript
function interpolate(pattern: string, values: Record<string, string>): string {
  return pattern.replace(/%(\w+)%/g, (_, key) => values[key] ?? `%${key}%`);
}
```

Uses `%name%` delimiter — avoids collisions with Vue `{{ }}`, Svelte `{ }`, and Kern `<<<>>>`.

### 4.4 renderConditional() — The Hardest Function

```typescript
function renderConditional(
  c: { test: string; then: AnalyzedChild[]; else?: AnalyzedChild[] },
  d: TargetDecisions,
  indent: number
): string {
  const pad = '  '.repeat(indent);

  switch (d.conditional.mode) {
    case 'wrap_block': {
      const lines = [interpolate(d.conditional.open, { test: c.test })];
      lines.push(...c.then.map(ch => renderChild(ch, d, indent + 1)));
      if (c.else) {
        lines.push(pad + d.conditional.elseOpen);
        lines.push(...c.else.map(ch => renderChild(ch, d, indent + 1)));
      }
      lines.push(pad + d.conditional.close);
      return lines.map(l => l.startsWith('  ') ? l : pad + l).join('\n');
    }

    case 'decorate_child': {
      const renderable = c.then.filter(ch => ch.kind === 'node');
      if (renderable.length === 1 && isDirectiveHost(renderable[0])) {
        // Single renderable child: add directive to it
        const node = { ...renderable[0].node };
        node.attrs = [
          ...node.attrs,
          { kind: 'static' as const, name: interpolate(d.conditional.attr, { test: c.test }), value: '' }
        ];
        let result = renderNode(node, d, indent);
        if (c.else) {
          const elseRenderable = c.else.filter(ch => ch.kind === 'node');
          if (elseRenderable.length === 1 && isDirectiveHost(elseRenderable[0])) {
            const elseNode = { ...elseRenderable[0].node };
            elseNode.attrs = [...elseNode.attrs, { kind: 'static' as const, name: d.conditional.elseAttr, value: '' }];
            result += '\n' + renderNode(elseNode, d, indent);
          }
        }
        return result;
      }
      // Multi-child: wrap in <template v-if>
      const lines = [`${pad}<${d.conditional.wrapperTag} ${interpolate(d.conditional.attr, { test: c.test })}>`];
      lines.push(...c.then.map(ch => renderChild(ch, d, indent + 1)));
      lines.push(`${pad}</${d.conditional.wrapperTag}>`);
      if (c.else) {
        lines.push(`${pad}<${d.conditional.wrapperTag} ${d.conditional.elseAttr}>`);
        lines.push(...c.else.map(ch => renderChild(ch, d, indent + 1)));
        lines.push(`${pad}</${d.conditional.wrapperTag}>`);
      }
      return lines.join('\n');
    }

    case 'expr_wrapper': {
      const thenBody = c.then.map(ch => renderChild(ch, d, indent + 1)).join('\n');
      if (c.else) {
        const elseBody = c.else.map(ch => renderChild(ch, d, indent + 1)).join('\n');
        return pad + interpolate(d.conditional.ternaryPattern, { test: c.test, then: thenBody, else: elseBody });
      }
      return pad + interpolate(d.conditional.andPattern, { test: c.test, body: thenBody });
    }
  }
}
```

~65 LOC. Handles Vue, Svelte, React. Includes Codex's countRenderableChildren/isDirectiveHost checks.

## 8. Open Questions

1. **Should U produce AnalyzedComponent as a new type (Option C) or enrich IRNode with metadata (Option H)?** Both work. H is simpler; C is more type-safe. Can start with H, migrate to C if needed.

2. **Should the TailwindClassifier be part of R or a separate plugin?** Currently modeled as plugin. Could be inlined into R's scoped_css → utility_classes style mode switch.

3. **How does UDR handle evolved nodes?** Evolved nodes add analysis rules to U + optional manifest entries to D. R renders automatically. Needs design in detail.

4. **Should manifests be runtime-loadable?** If yes: community targets as npm packages (`npm install @kern-community/angular`). If no: all manifests bundled at build time.
